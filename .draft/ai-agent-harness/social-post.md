# AI Agent 工程的残酷真相：你搭的东西，注定要被拆掉

Manus 自 2024 年 3 月以来，已经重构了 5 次。

LangChain 的 Open Deep Research，一年内重建多次。Anthropic 也在持续拆掉 Claude Code 外面的 Agent Harness。

这不是工程失败。

如果你今天在做 AI 系统，这反而更像一种健康信号。

因为真正变化的，不只是模型变强了，而是工程问题本身已经换了。

---

过去，大家比的是谁能训练出更强的模型。

现在，越来越多团队拿到的是同一种东西：基础模型 API。

也就是说，你手里拿到的，不再只是一个需要继续优化的模型，而是一个极其强大、但也并不稳定的新型计算原语。

于是核心挑战变成了另一件事：

**不是“怎么训练模型”，而是“怎么把模型编排成一个可靠系统”。**

Prompt Engineering、Context Engineering、Tool Orchestration、Evaluation、Human-in-the-loop、Agent 设计，这些都因此从边缘工作变成了主战场。

但这里真正值得记住的，不是这些名词本身。

而是背后那条更底层的判断：

**模型越强，好的系统往往越简单。**

---

这也是 Lance Martin 这轮讨论里，最值得带走的一句话。

Rich Sutton 在《The Bitter Lesson》里讲过，最终胜出的，往往不是那些手工雕刻得最精巧的系统，而是更通用、更能持续吃到计算红利的方法。

今天，这个教训已经来到应用层。

你加在模型外面的很多结构，背后其实都藏着一句潜台词：

“模型现在还不擅长 X，所以我先补一层 Y。”

问题在于，这种判断往往只有几个月寿命。

一旦模型变强，原来那层 Y 就不再是辅助，反而会变成摩擦、噪音，甚至新的瓶颈。

这层包在 LLM 外面的脚手架，就是很多人说的 **Agent Harness**。

它负责工具调用、消息历史、上下文管理、错误恢复，以及模型和外部世界之间的大部分接口。

它当然重要。

但它也最容易被过度设计。

所以今天做 AI 工程，一个非常反直觉但很现实的原则是：

**不要默认系统会越做越厚。更常见的情况是，模型每前进一步，Harness 就该再薄一点。**

这也是为什么 Manus 会在短时间里多次重构，Open Deep Research 会反复重建，Anthropic 也会持续拆脚手架。

不是第一版写得差。

而是底层能力变了，旧结构开始妨碍它继续变好。

---

很多团队一上来就想做 Agent。

但真正的问题，往往比“要不要 Agent”更早一步：

如果你连“什么叫做对了”都定义不清楚，Agent 只会把混乱自动化。

这也是为什么，在这篇讨论里，另一个被低估的核心，不是“自治”，而是 **Evaluation**。

LLM 不是传统软件。

它不会每次都给你同样的输出。

所以你不能只靠 prompt 手感和 demo 直觉来判断系统有没有进步。

你得能说清楚：

什么场景它稳定。

什么错误可以接受。

什么错误绝对不行。

什么指标真的在变好。

这里有个非常关键的判断，叫 **Verifier's Law**：

**一个任务越容易验证，AI 就越容易在这个任务上被训练、被约束、被优化。**

这句话的潜台词是，很多系统做不起来，不一定是模型不够强，而是你根本没有把“正确”定义清楚。

所以别急着上复杂 Agent。

先把 verifier 搭起来，很多问题才会真正变得可工程化。

---

但问题还没有结束。

就算 verifier 有了，Agent 系统最容易爆掉的，往往还是上下文。

很多人的默认做法，是把所有工具调用结果都无脑塞回消息历史。

短期看，这很省事。

长期看，它几乎一定失控：

上下文越来越长。

成本越来越高。

延迟越来越大。

模型越来越不听话。

这就是很多人说的 **Context Rot**。

上下文不是越长越强。

很多时候，越长，只是越乱。

所以真正有效的 Context Engineering，核心不是“再塞更多信息”，而是三件事：

**1. Reduce**

主动压缩上下文。

旧工具调用不必原文全带，保留摘要就够了。

历史太长时，要做轨迹总结，而不是无限追加。

**2. Offload**

把不该放在 prompt 里的东西搬出去。

完整工具结果放到外部存储，需要时再取。

更重要的是，把动作空间也搬出去。

与其给 Agent 塞 100 个工具说明，不如给它几个原子工具，比如 bash terminal。

这样能力边界更大，prompt 反而更干净。

**3. Isolate**

把重任务交给子 Agent，在隔离上下文里完成。

主 Agent 不需要背着全部历史思考，只需要接收精炼结果。

一句话总结：

**主 Agent 不该背着整个世界思考。**

能压缩的压缩。

能外包的外包。

能隔离的隔离。

---

这也顺手解释了另一个常见误区：

**不是所有任务都需要 Agent。**

Workflow 和 Agent 不是非此即彼，而是一条连续光谱。

Workflow 适合步骤已知、结构稳定、追求可预测性的任务。

比如跑测试、迁移代码、固定流程的文档处理。

Agent 适合路径未知、需要探索、需要临场决策的任务。

比如开放式调研、复杂排障、长链路编码。

很多真正好用的系统，实际上不是“全 Agent”，而是“在 Workflow 里嵌一个 Agent”。

所以问题从来不是“要不要 Agent”，而是：

**这件事，到底需要多大自主性？**

如果任务高风险、反馈慢、验证难，就不要轻易放权。

如果任务低风险、反馈快、有人监督，Agent 才更容易释放价值。

---

如果把整篇文章压成几条真正能带走的原则，我会记住这五句：

1. **先从简单开始。** 先把 prompt 和 workflow 用到极致，再考虑 Agent。
2. **为变化而设计。** 不要假设今天的模型短板会长期存在。
3. **把评估放在前面。** 没有 verifier，就没有可靠优化。
4. **把上下文当成稀缺资源。** Reduce、Offload、Isolate 不是技巧，而是系统的生存条件。
5. **把重建当成常态。** 真正的能力，不是第一次搭出来，而是下一次能更快拆掉重搭。

---

当然，这里还有一个边界。

“持续重建”不是让所有生产系统都进入永久动荡。

高稳定、高合规、高可靠场景，依然需要更稳的边界、更强的测试、更慢的变更节奏。

但即便如此，方向还是很清楚：

**AI 应用时代，真正的护城河，越来越不是模型本身，而是围绕模型的编排能力。**

再说得更直白一点：

**不是你能把系统搭得多复杂，而是你能不能随着模型变强，持续删掉那些已经不再必要的复杂度。**

这可能才是今天 AI Agent 工程里，最反直觉、也最残酷的真相。
