# Cursor Subagents 文章架构设计：多视角综合分析

> 三个独立视角（x-claude / x-gemini / x-gpt）的综合产物。
> 每个维度取最强方案，冲突处以推理质量裁定。

---

## 0. 设计总纲（一句话）

**Subagent 不是"更多的 prompt"，而是把开发者脑中隐性的"调度器"外化为可版本化的团队架构：职责边界（context isolation）+ 并行（parallel）+ 路由（auto delegation）+ 文件即定义（file as agent）。**

---

## 1. 叙事弧线（Narrative Arc）

### 策略：从崩溃现场反推架构

不从"什么是 subagent"起笔，而是让读者在解决真实问题的过程中**自行推导出** subagent 的必要条件——然后发现 Cursor 已经实现了它。

### 五幕结构

```
崩溃 → 诊断 → 封死退路 → 重新发明 → 发现 Cursor 已实现 → 实战验证 → 升华
```

| 幕 | 叙事功能 | 读者心理状态 |
|---|---------|-----------|
| **第一幕：崩溃现场** | 建立共鸣——所有 Cursor 用户都经历过的 context 崩塌 | "对，我遇到过这个！" |
| **第二幕：诊断** | 归因——不是模型不行，是单 agent 架构有天花板 | "原来是架构问题？" |
| **第三幕：三条死路** | 排除法——加大 context、串行化、多窗口全部失败 | "那怎么办？" |
| **第四幕：重新发明** | 读者自行推导——如果 agent 能 fork 自己呢？ | "等等，我想到了！" |
| **第五幕：发现 + 实战** | 揭示 Cursor 的实现 + 立刻上手验证 | "这也太简单了吧" |
| **尾声：升华** | 文件夹即团队——从工具到组织架构的认知跃迁 | "这改变了我的工作方式" |

### 开场钩子（具体场景）

你让 Cursor agent 同时重构一个模块、写测试、更新文档。第 10 轮交互后，它开始"忘记"之前改了什么——测试引用了已删除的函数，文档描述了不存在的接口。你开始怀疑：是模型变笨了吗？

**承诺**：本文不从"什么是 subagent"开始，而从一个问题开始——**为什么你明明有 AI，却仍然像单线程一样工作？**

## 2. 核心隐喻选择

### 三个候选方案评估

| 隐喻 | 独立上下文 | 并行执行 | 自动委派 | 文件即定义 | 叙事温度 | 技术精度 |
|------|---------|---------|---------|----------|---------|---------|
| A. 厨房旅制度 | ★★★ 独立备料台 | ★★★ 多工位同时烹饪 | ★★★ 主厨按菜单分派 | ★★★ 食谱卡=agent | ★★★ 故事性极强 | ★★ 需补技术锚 |
| B. 电影剧组 | ★★ 各部门独立 | ★★★ 多部门同时开工 | ★★★ 副导演按类型派活 | ★★ 工作手册=agent | ★★★ 直觉友好 | ★★ 需补技术锚 |
| C. OS 进程模型 | ★★★ 独立内存空间 | ★★★ 多核并行 | ★★ 调度器路由 | ★ 映射较弱 | ★ 冷硬 | ★★★ 完美精确 |

### 推荐：A（厨房旅制度）主隐喻 + C（OS fork）技术锚定

**理由**：

1. **厨房旅制度**提供叙事骨架——Escoffier 在 19 世纪发明了这个系统，解决的是同一个问题：一个主厨不可能同时炒所有菜。每个工位有独立备料台（context isolation），主厨看菜单自动分派（auto-delegation），所有工位同时运作（parallel），每个工位的操作手册就是一张食谱卡（file as agent）。
2. **OS fork()** 在关键技术点做一句话精确锚定，确保技术读者也能把 subagent 映射到已有心智模型。

**具体用法示例**：

> 每个 subagent 拥有独立的 context window——如果你熟悉 Unix，这就是 `fork()`：子进程有自己的内存空间，不会污染父进程。如果你更喜欢实体隐喻：这是厨房旅制度中每个工位的独立备料台——鱼档的腌料不会溅到甜点档的奶油上。

### 为什么不选电影剧组

电影剧组（x-gpt 推荐）也是好隐喻，但有两个弱点：
1. 太常见，缺乏"发现感"——读者看到第一秒就知道你要说什么
2. "上下文隔离"在剧组隐喻中不够锐利（各部门确实会共享信息）

厨房旅制度更锐利：每个工位**物理上**被分隔，备料台**不可能**共用，这与 context isolation 的硬隔离特性完美对应。

## 3. Aha Moments 设计（4 个，递进关系）

| # | 位置 | 洞见 | 传达方式 |
|---|------|------|---------|
| **Aha 1** | §1 末尾 | **不是模型不够强，是"一个 agent"这个架构有天花板** | context 物理限制 + 白板类比 |
| **Aha 2** | §2→§3 过渡 | **如果 agent 能 fork 自己呢？**（读者自行推导出 subagent） | 纯文字逻辑推演，不用辅助组件 |
| **Aha 3** | §4 开头 | **一个 Markdown 文件就是一个完整的 Agent。没有第二个文件。** | 展示 15 行 .md 文件 + 一句话点破 |
| **Aha 4** | §6 结尾 | **文件夹就是团队——agent 团队可以 git 管理、版本化、code review** | 短而锐的概念升华 |

**补充洞见**（来自 x-gpt，嵌入 §4 而非独立 aha）：

> `description` 不是介绍，是**路由规则**——它是写给系统看的职责边界，不是写给人看的自我介绍。

这个洞见足够锐利，但不需要独立为第 5 个 aha——嵌入 §4 的 YAML 字段讲解中作为"反直觉提醒"即可。

## 4. 认知节奏设计

### 节奏原则

- **推力段与呼吸段交替，但呼吸段也携带信息**（用类比/图示/小结编码）
- **高密度段只引入一个新概念**，立刻给出可复述的一句结论
- **三种摩擦框架**贯穿全文（上下文污染 / 等待阻塞 / 人工路由），每个特性都回答"它减少了哪种摩擦"

### 各章节密度与节奏

| 章节 | 密度 | 节奏 | 呼吸手段 | 建议字数 |
|------|-----|------|---------|---------|
| §0 崩溃现场 | 中 | 快→停顿 | 场景叙事本身即呼吸 | 500-600 |
| §1 一块白板的极限 | 高 | 逐步加压 | **SVG 图 1** | 800-900 |
| §2 三条死路 | 高 | 快速排除→悬念 | 对比表格 | 600-700 |
| §3 如果主厨能分身 | 中 | 舒展，叙事 | **SVG 图 2** + 类比展开 | 700-800 |
| §4 一个文件的全部 | 高 | 逐层揭示 | 代码块 + note 卡片交替；**SVG 图 3** | 900-1000 |
| §5 实战 | 中 | 匀速，手把手 | 代码块自然呼吸；**SVG 图 4** | 800-900 |
| §6 文件夹即团队 | 低 | 缓，收束 | 短洞察 + 开放思考 | 400-500 |

**总字数**：~4700-5400 字

### 关键节奏控制点

- **§1→§2**：全文第一个加压峰值。读者意识到问题不可调和。
- **§2 末尾→§3 开头**：全文最关键的节奏转换——从"压力最大"到"忽然打开"。需要一个视觉分隔（`<hr>`），然后用轻松的厨房故事接住读者。
- **§4**：信息密度最高，需要代码块和 note 卡片频繁交替调节。
- **§5**：刻意控制匀速——不加速不减速，让读者感觉"跟着做就行"。

## 5. 实战例子设计

### 推荐方案：PR Review 双人组

**场景**：你提交了一段代码变更，希望同时获得安全审查和性能审查。

### 为什么选这个

| 维度 | 评估 |
|------|------|
| **最小化** | 只需 2 个 .md 文件，每个 < 20 行 |
| **展示力** | 并行执行 ✓ 自动委派 ✓ 独立上下文 ✓ readonly ✓ |
| **实际用途** | 每个开发者每天都在做 code review |
| **5 分钟可复现** | 创建 2 个文件 → 一句指令触发 |

### 为什么不选其他方案

| 方案 | 淘汰原因 |
|------|---------|
| Repo Quick Triage（3 agents） | 更有力但更复杂，不适合"第一个"工作流 |
| 全栈开发团队 | 太大，5 分钟复现不了 |
| 代码翻译 | 单 agent 就能做，体现不出并行价值 |
| 文档生成 | 太无聊，缺乏冲击力 |

### 具体实现

**文件 1**：`.cursor/agents/security-reviewer.md`

```yaml
---
name: security-reviewer
description: Reviews code changes for security vulnerabilities, injection risks, secret leaks, and unsafe patterns.
model: fast
readonly: true
---
```

```markdown
You are a security-focused code reviewer. For any code change:
1. Check for injection vulnerabilities (SQL, XSS, command injection)
2. Identify hardcoded secrets or credentials
3. Flag unsafe deserialization or eval usage
4. Check dependency security (known CVEs)

Output: ranked findings with severity (Critical / High / Medium / Low).
If clean, state "No security issues identified."
```

**文件 2**：`.cursor/agents/perf-reviewer.md`

```yaml
---
name: perf-reviewer
description: Reviews code changes for performance issues, N+1 queries, memory leaks, and unnecessary computations.
model: fast
readonly: true
---
```

```markdown
You are a performance-focused code reviewer. For any code change:
1. Identify N+1 query patterns or unnecessary database calls
2. Flag potential memory leaks or unbounded growth
3. Check for O(n²) or worse algorithms where O(n) is possible
4. Spot unnecessary re-renders or redundant computations

Output: specific findings with estimated impact (High / Medium / Low).
```

### 触发方式

**自动委派**（主推）：

> "帮我 review 一下这个 PR 的改动"

主 agent 读取两个 agent 的 description，自动同时派发。两者并行运行、各自返回报告。

**显式调用**（对照展示）：

> `/security-reviewer` 检查这个文件有没有注入风险

### 进阶提示（可选，用 aside 组件）

> 试试加入第三个 agent `docs-checker.md`，让它检查文档是否与代码变更同步。
> 恭喜你——你的 review 团队从 2 人变成了 3 人，只需要多创建一个文件。

## 6. SVG 图设计清单（4 张）

### 图 1：一块白板的极限（§1 插入）

- **标题**：Context Window 的物理极限
- **核心概念**：单 agent context 随任务增加逐渐溢出、互相覆盖
- **视觉设计**：
  - 一个矩形"白板"，内部分区填充不同颜色

    - 绿 = 测试上下文

  - 最后一个区域用红色虚线标"溢出"

  - 顶部：Main Agent 节点（深色）

- **标题**：Anatomy of a Subagent

- **视觉设计**：
  - 左侧：.md 文件可视化（YAML header + markdown body）
  - 右侧：运行时概念，用引线连接

## 7. 完整章节大纲

---

### §0 — 崩溃现场

**副标题**：当一个 Agent 试图成为所有人

**核心论点**：单 agent 在复杂任务中的崩溃不是能力问题，是架构问题。

**内容要点**：
- 开场钩子：具体场景——重构 + 测试 + 文档同时进行，第 10 轮交互后 agent 开始自相矛盾
- 三种摩擦的具象化：上下文互相污染、等待时间不可折叠、你成了人工路由器
- "如果你用 Cursor 超过一周，你一定遇到过这个时刻"

**长度**：500-600 字 | **密度**：中

---

### §1 — 一块白板的极限

**副标题**：Context Window 不是越大越好

**核心论点**：单 agent 的 context window 是有限的共享资源，多任务并行时必然互相挤压——这是物理限制，不是工程缺陷。

**内容要点**：
- "白板"类比：一个人在一块白板上同时画三张图，互相覆盖是必然结果
- 即使 128K token context，同时容纳代码 + 测试 + 文档上下文时，有效注意力急剧下降
- **插入 SVG 图 1**
- **Aha #1**：不是 agent 不够强，是"一个 agent"这个架构有天花板

**长度**：800-900 字 | **密度**：高

---

### §2 — 三条死路

**副标题**：加大、串行、多窗口——全部破产

**核心论点**：面对 context 瓶颈，直觉上的三种解决方案全部行不通。

**内容要点**：
- **退路 1**：加大 context window → 注意力稀释、成本指数增长
- **退路 2**：拆成顺序步骤 → 丧失并行性，3x 时间
- **退路 3**：开多个 chat 窗口 → 失去协调性，各干各的
- 用对比表格/卡片展示三种方案的失败
- 悬念收束："三条路都不通——但还有第四条"

**长度**：600-700 字 | **密度**：高

---

### §3 — 如果主厨能分身

**副标题**：从厨房旅制度到 Subagent 的诞生

**核心论点**：subagent 的核心设计——独立上下文 + 并行执行 + 自动委派——与 Escoffier 的厨房旅制度完全同构。

**内容要点**：
- Escoffier 的故事：19 世纪面对同一个问题——一个主厨不可能同时炒所有菜
- 厨房旅制度映射表：主厨=主 agent，工位=subagent，备料台=context，菜单分派=auto-delegation
- fork() 精确锚定："子进程有独立内存空间，不会污染父进程"
- **插入 SVG 图 2**
- **Aha #2**：读者自行抵达"分裂"的解决方案

**长度**：700-800 字 | **密度**：中（§1-§2 高压后的呼吸段）

---

### §4 — 一个 Markdown 文件的全部

**副标题**：Subagent 的解剖学

**核心论点**：Cursor subagent 的定义极其简洁——一个 .md 文件 = YAML 配置 + Markdown 指令，这就是全部。

**内容要点**：
- 展示完整 .md 文件，逐字段解释（但每个字段回答"它减少了哪种摩擦"）
  - `name`：稳定调用接口
  - `description`：**路由规则，不是自我介绍**（反直觉提醒）
  - `model`：预算与能力的选择
  - `readonly`：安全边界
  - `is_background`：承认现实——有些事就是慢
- 自动委派机制：主 agent 如何读取 description 决定路由
- 显式调用：`/agent-name` 语法
- 内置 subagent：explore、bash、browser
- **插入 SVG 图 3**
- **Aha #3**："一个 Markdown 文件就是一个完整的 agent。没有第二个文件。"
- note 卡片：对比 LangChain/CrewAI 需要的代码量 vs 一个 .md 文件

**重要澄清**（来自 x-gemini 的 blindspot 提醒）：
> ⚠️ Subagent ≠ Cursor Rules。Rules 是全局被动约束（Constitution），Subagent 是按需唤醒的主动执行者（Worker）。

**长度**：900-1000 字 | **密度**：高（全文峰值，用代码块和 note 卡片调节）

### §5 — 5 分钟，一个 Review 团队

**副标题**：从零构建你的第一个多 Agent 工作流

**核心论点**：用 5 分钟建一个能并行工作的 PR review 团队。

**内容要点**：
- Step 0：创建 `.cursor/agents/` 目录
- Step 1：写 `security-reviewer.md`（完整代码）
- Step 2：写 `perf-reviewer.md`（完整代码）
- Step 3：触发——展示自动委派 + 显式调用两种方式
- **插入 SVG 图 4**：执行链路时间线
- 预期结果：两个 reviewer 并行返回各自审查报告
- 进阶提示：加入 `docs-checker.md` 扩展为三人组

**长度**：800-900 字 | **密度**：中（匀速手把手）

---

### §6 — 文件夹即团队

**副标题**：当组织架构变成代码

**核心论点**：`.cursor/agents/` 不是配置目录，而是可 git 管理的团队组织架构。

**内容要点**：
- **Aha #4**：文件夹 = 团队的组织洞察
- 推论 1：团队可 git 版本化——`git blame` agent 定义的演变
- 推论 2：团队可跨项目 fork——复制别人的 `agents/` 就有了他们的团队
- 推论 3：团队可 code review——PR 中 review agent 定义变更
- 从"一个文件 = 一个 agent"到"一个文件夹 = 一个团队"到"一个仓库 = 一个组织"
- 结尾不写"总结"，让文章在洞察中自然结束

**长度**：400-500 字 | **密度**：低（收束，让洞察沉淀）

## 8. 三视角对比与裁定记录

### 共识点（三个视角一致）

1. **叙事方式**：从痛点出发 → 读者自行推导 → 发现 Cursor 已实现（不平铺直叙）
2. **核心 aha**：单 agent 天花板 → file = agent → auto-delegation 的路由本质
3. **SVG 数量**：3-4 张，覆盖 context 问题 + 执行模型 + 文件解剖
4. **实战要求**：小（5 分钟）、有力（展示并行）、有实际用途

### 分歧点与裁定

| 分歧 | x-claude | x-gemini | x-gpt | 裁定 |
|------|---------|---------|------|------|
| **主隐喻** | 厨房旅制度 | OS 进程/线程 | 电影剧组 | **采用 x-claude**：叙事温度最高，且物理隔离的映射最精确 |
| **辅助隐喻** | Unix fork() | Unix 哲学 | OS 进程 | **采用 x-claude**：fork() 一句话锚定即可 |

### 各视角独特贡献

- **x-claude**：厨房旅制度隐喻（叙事骨架）、"文件夹即团队"的 git 管理洞察（§6 升华）、精确的节奏控制分析
- **x-gemini**：Rules vs Subagent 的必要澄清（防止读者混淆）、"上下文隔离 = 提升智商"的锐利表述
- **x-gpt**：`description = 路由规则`的反直觉提醒、"三种摩擦"框架（上下文/等待/路由）、"字段不是百科词条而是摩擦解决器"的写作纪律

## 9. 写作纪律（Anti-Patterns 清单）

1. **不滑回字段说明书**：每个字段/特性都必须回答"它减少了哪种摩擦？"——否则删掉
2. **不展示"思考过程"**：输出 Diamond，不输出 Mining Equipment
3. **不空洞过渡**：呼吸段也携带信息（用类比/图示编码）
4. **不写"总结"章节**：让文章在洞察中自然结束
5. **不过度隐喻**：厨房旅制度用在 §3（推导）和 §6（收束），其他章节不反复提及

## 10. 风险与缓解

| 风险 | 概率 | 缓解方案 |
|------|------|---------|
| 厨房隐喻对部分读者"空转"（没有餐饮经验） | 中 | 每次使用旅制度隐喻后，立刻用 fork() 做技术锚定 |
| 读者混淆 Subagent 与 Cursor Rules | 高 | §4 中用 note 卡片明确区分 |
| §4 信息密度过高导致读者疲劳 | 中 | 代码块 + note 卡片交替；每个字段用"解决什么摩擦"而非"是什么" |
| 读者已了解 multi-agent 概念，觉得 §0-§2 啰嗦 | 低 | §0-§2 合计 ~1800 字，节奏快，不拖沓 |
