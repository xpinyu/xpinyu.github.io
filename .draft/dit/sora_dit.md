- 扩散模型面对的原始问题是：如何从纯噪声出发，稳定地得到一张或一段符合条件的内容。
- 它的基本做法不是直接“画”内容，而是学习在不同时间步上**识别并预测当前样本中的噪声成分**。

## 核心观点一：扩散模型学的不是“画图”，而是“认噪声”

**扩散模型最小可理解单位，是：给它一张被污染到任意程度的图，它去预测里面的噪声。**

讲者用一个很形象的例子讲这个过程：从一张猫的图开始，时间步 \(t=0\) 时是原始图像；随后不断加一点噪声、再加一点噪声，直到最后变成“100% 纯噪声”。他把这个过程理解为一个信噪比不断下降的插值：左边是 100% 信号，右边是 100% 噪声，中间每一步都是两者的混合。

这个图景很重要，因为它纠正了一个常见误解。很多人直觉上会觉得，扩散模型是在学习“如何从无到有生成图像”。但按讲者的解释，更准确的说法是：**它在学习一个反问题——看到某个噪声等级下的样本，判断其中的噪声是什么。**

> 你把一张带噪图像送进模型，模型输出的是“这张图里有哪些噪声”，然后你把这些噪声减掉，图像就更干净一点。

这也解释了为什么采样是迭代式的。讲者用了一个“沿曲线走路”的比喻：如果真实的数据流形是弯曲的，你从噪声点一下子跨到终点，误差会很大；所以更合理的方式是每一步都只朝模型建议的方向挪一点。这就是为什么扩散采样看起来像一个 ODE/SDE 求解过程，而不是一次前向就结束。

**为什么这不直观？**  
因为“生成”这个词让人误以为模型直接输出目标。可扩散模型真正稳定的地方，反而来自它把任务拆成了很多个局部子任务：每次只判断“这里的噪声是什么”。

**这意味着什么？**  
一旦你接受了“预测噪声”这个视角，后面很多设计就都变得更顺了：为什么要有时间步编码，为什么可以做条件控制，为什么 latent 空间更高效，为什么 Transformer 只需要接过“噪声预测器”的角色，而不需要发明一整套新生成范式。

## 核心观点二：真正让扩散可用的，不只是去噪，而是先离开像素空间

**如果直接在像素上做扩散，成本高得离谱；latent diffusion 的关键，是先把图像压缩到一个更小但仍保留语义的空间。**

讲者专门花了不少篇幅解释 VAE 在这里的角色，因为这是从“理论可行”到“工程上真的能跑”的关键台阶。直观上，如果一张图是 256×256×3，模型每一步都要在这个空间里做迭代去噪，计算量非常大。更糟的是，扩散不是一次计算，而是很多步。

他给了一个非常具体的对比：Stable Diffusion 里，图像可能从 **256×256×3 压到 32×32×4**。这不是小优化，而是维度级别的缩减。于是扩散过程不再发生在原始像素上，而发生在这个 latent 上。

这个压缩过程由一个先训练好的 VAE 提供：编码器把图像压成 latent，解码器再把 latent 还原回图像。扩散模型训练时，真正去加噪、去预测噪声的对象，是 latent 表示，而不是原始像素。

具体图景可以想成这样：先把“猫”的图像送进编码器，得到一个压缩后的 latent；然后对 latent 做和前面同样的加噪链条；模型学习在这个 latent 空间里预测噪声；最后从去噪后的 latent 出发，用冻结的解码器还原成图像。

**为什么这不明显？**  
因为很多人第一次接触扩散时，脑中图景还是“在图片本身上加噪和去噪”。但现代高质量系统里，更常见的是“在有语义压缩的表示空间上做扩散”。这一步看似是工程技巧，实际上改的是问题难度。

**这意味着什么？**  
它让扩散模型从“概念上优雅，但太慢太贵”，变成“可以在大规模数据和更大网络上训练”的系统。没有 latent diffusion，后面的 DiT 很难成为现实中的主力方案。

## 核心观点三：DiT 的真正更新，不是“用了 Transformer”这么简单，而是把图像改写成 token 序列

**DiT 的核心动作，是把 latent 图像切成 patch，再像 ViT 一样把每个 patch 当成 token，送入 Transformer 预测噪声。**

讲者的语气里有一点玩笑：“他们把常用 U-Net 换成 Transformer，发现——哇，它居然很能扩展。显然，Transformer 本来就很能扩展。”这个玩笑其实点中了重点：DiT 的价值，不只是“替换了一个骨干网络”，而是把扩散模型放进了今天最成熟的规模化范式里。

具体怎么做？讲者从 patchify 讲起。你把一张 latent 图像切成很多小块，每个 patch 展平后变成一个 token。第一个 patch 对应第一个 token，第二个 patch 对应第二个 token……然后给这些 token 加上位置编码，再送入标准 Transformer。

这一步最关键的启发是：**图像不再被模型当成一张连续二维网格去处理，而是被重新组织成一串离散 token。** 只要这串 token 的表示足够合理，Transformer 就能处理它们之间的关系。

讲者也仔细区分了几种条件注入方式。最朴素的方法是把条件信息——比如时间步、类别、文本 embedding——直接拼到序列里，但他明确说这种“in-context 拼接”效果不太好。更常见、更有效的方法是：
1. 用交叉注意力处理高信息量条件，比如文本；
2. 用 adaptive layer norm 之类的方式，把时间步、类别等较小向量注入到网络层里。

他特别解释了 adaptive layer norm 的直觉：普通 layer norm 会把特征规范到某个标准分布；而 adaptive 版本允许**条件信息来决定这个规范化后的平移和缩放参数**。也就是说，条件不是简单贴在输入旁边，而是直接参与塑造特征分布。

> 这相当于让条件信息决定：这一层里哪些特征应该被放大，哪些应该被压低。

他还提到 DiT 论文里的一个有趣细节：做 classifier-free guidance 时，他们在某些实验里**只对 latent 的前 3 个通道施加 guidance，而不是全部 4 个通道**，效果几乎差不多。这类细节说明，条件控制并不一定要粗暴地作用于所有表示维度。

**为什么这不明显？**  
因为“把 U-Net 换成 Transformer”听上去像结构替换，但真正改变可扩展性的，是 token 化表示 + Transformer 的规模律。不是任何替换都能带来这种效果，关键在于问题被改写成了 Transformer 擅长的形式。

**这意味着什么？**  
当任务被改写成 token 序列建模后，模型容量、训练数据、计算规模更容易持续加上去。讲者多次强调“scale is all you need”虽然带点调侃，但对这类系统确实抓住了主要矛盾：一旦架构吃规模，顶级公司的优势就会迅速放大。

## 核心观点四：Sora 最重要的延伸，是把“图像 patch token”变成“视频时空 token”

**Sora 在公开信息里最值得关注的，不只是用了 DiT，而是把 patch/token 思路扩展到了视频的时空表示。**

这里讲者很谨慎。他承认没有论文，只有技术报告，所以很多具体实现只能推测。但从公开描述里，他认为最合理的理解是：先用视频压缩网络把视频编码成 latent，再把这些 latent 切成**时空 patch**，拼成一个长序列，交给 Transformer 做去噪。

他给出了一个非常清晰的想象图：如果图像是“一张图切成很多 patch token”，那视频就是“每一帧切出 patch token，然后把多帧 token 串起来”。于是 Transformer 处理的，不再是单张图中的 patch 关系，而是**整个视频里的 patch 关系**。每个 token 代表的，是视频某个时空位置的一小块内容。

这个表示法为什么关键？因为视频最难的问题不是单帧细节，而是**时间一致性**。当同一个模型在统一序列里同时看到多个时间片段，它至少在表示层面拥有了处理跨帧依赖的机会。

讲者还进一步推测，Sora 很可能不只需要一种位置编码。因为视频里有至少三种位置信息：
1. patch 在画面中的横向位置；
2. patch 在画面中的纵向位置；
3. patch 属于视频中的第几帧。

如果还要支持不同长宽比、不同视频时长，位置编码设计会更复杂。他猜测这可能需要多组位置编码，而不是简单的一维顺序编码。这里他说得很坦白：这是根据报告描述做的合理想象，不是已公开确认的实现。

一个很重要的旁支是可变长宽比。讲者指出，Sora 技术报告特别提到不同持续时间和不同 aspect ratio，这意味着模型不是被迫把所有内容硬塞进固定方形窗口里。这个点看似细小，但它直接关系到生成结果会不会被奇怪裁切。

**为什么这不明显？**  
因为很多人看到 Sora，会把注意力放在“它终于能生成更真实的视频”。但从建模角度看，更深一层的更新其实是：视频不再只是“很多图片逐帧生成”，而是被作为一个统一的时空 token 场来建模。

**这意味着什么？**  
这给了模型一个更自然的路径去学习对象持续性、摄像机运动、场景几何和事件延续。它未必自动解决一切一致性问题，但至少把问题放到了一个更适合全局建模的表示中。

## 核心观点五：Sora 的能力不只来自主干网络，还来自“怎么给模型喂条件”

**高质量生成不只是骨干模型更大，还包括条件信息是否足够清晰、足够结构化。**

讲者在 Sora 部分专门提到两个条件侧的关键做法。

第一个是 **prompt 重写与重描述**。他提到 DALL·E 3 的经验：人类写的提示词往往太短、太模糊，所以他们先训练模型对图像做更好的 caption，再用这些更详细的描述训练生成模型；同时，用户给出的短 prompt 还会被 GPT 扩写成更细致的长描述。Sora 技术报告里也提到类似做法。

这意味着，模型变强不只是因为“会画”，也因为训练和推理时接收到的语言条件更明确。讲者认为，这会带来一种很直接的体验差异：**prompt 跟随能力更强**。他说如果你用过 DALL·E 3，通常会明显感觉它比 Stable Diffusion 1.5 更懂你的描述。

第二个是 **mask 条件生成**。这部分他借用了 Lumiere 的思路来帮助理解：如果你不是从零生成整段视频，而是希望“给一帧图，让它动起来”，或“保留前几帧，往后续写”，那就可以把输入拆成三部分：
- 已知内容；
- 需要生成的位置上的噪声；
- 一个 mask，告诉模型哪里该保持、哪里该续写。

这个机制很强，因为它不是让模型“自己猜哪里不能改”，而是显式告诉它。讲者用它解释了 Sora 可能支持的几类能力：图生视频、视频延长、视频编辑、视频拼接。

他还解释了视频编辑的基本逻辑：如果你有一段现成视频，不必把它完全打碎。你可以只往中间噪声程度推进一部分，再在新的文本条件下去噪回来。噪声加得越多，生成结果越偏离原视频；加得越少，保留的原结构越多。这和图像编辑中的 img2img 是同一个原理，只不过对象从单张图扩展成了视频。

**为什么这不明显？**  
因为看到惊艳 demo 时，人们很容易把全部功劳都归给“模型更大”。但在扩散系统里，条件工程常常不是边角料，而是决定“能不能按你想的方式生成”的一大半原因。

**这意味着什么？**  
如果 Sora 确实把更强的语言重写、mask 控制、时空 patch 建模结合起来，那么它强的不只是“随机生成一段漂亮视频”，而是更接近一个通用视频生成与编辑引擎。

## 连接起来看

把这些点连起来，Sora 的路线就变得清楚了：扩散模型提供了稳定的“逐步去噪”框架，latent diffusion 把问题压缩到可训练的表示空间，DiT 再把这个表示空间改写成 Transformer 擅长的 token 序列，而视频建模则把 token 从二维图像扩展到三维时空结构。讲者真正想传递的，不是某一个秘密技巧，而是一个很符合近年 AI 发展规律的判断：**当表示方式、架构和规模彼此对齐时，能力会出现非常陡的跃迁。Sora 看起来像跳变，底层却可能是一条连续的技术积累线。**

## 最小必要心智模型
- **建模的对象是什么：** 一个被噪声污染的图像或视频 latent，以及描述它应该长成什么样的条件信息。
- **它如何运动：** 模型在每个时间步预测当前样本中的噪声，迭代去噪；DiT 用 patch token + Transformer 来完成这件事，并且能靠规模继续变强。
- **它在哪里会失效：** 我们对 Sora 的很多具体实现仍不知道；而且高质量结果不只依赖架构，还依赖数据、算力、条件设计和视频表示方式，不能把成功简单归因于“用了 Transformer”。

## 一句话记住
Sora 最值得注意的，不是它“会生成视频”，而是它把扩散模型改写成了一个能在时空 token 上随规模增长的 Transformer 系统。

你这句话里已经有第一个误解了。  
扩散模型最核心的学习目标，不是“从无到有画图”，而是——材料里讲者反复强调的——**在任意噪声等级下，预测样本里的噪声是什么。**

他拿一张猫图举例：  
最开始是清晰的猫，接着一步步加噪，直到最后变成纯噪声。训练时，模型看到的是某个中间状态，比如“半脏不脏的一张猫”，它要做的不是直接输出一只猫，而是说：**这里面的噪声成分是什么。**

**我：**  
这区别真有那么大？最后不还是为了得到猫吗？

**苏格拉底：**  
有，而且很大。  
因为“直接画出猫”是一个大任务；“这一刻图里噪声是什么”是很多个小任务。

想象你在擦一面起雾的玻璃。  
“生成一只猫”听起来像你凭空画一只。  
“预测噪声”更像你在每一步判断：这层雾该去掉多少。  
扩散模型之所以稳，恰恰是因为它把生成拆成了很多局部判断。

材料里讲者甚至用了一个弯曲轨迹的比喻：  
如果真实数据分布像一条弯路，你从纯噪声一下跳到终点，容易跳歪；所以更合理的是一步步沿着模型给的方向挪。这也是为什么采样像 ODE/SDE 求解，不是一脚油门冲过去。

**我：**  
好，我先接受：扩散不是直接画图，而是在不断认噪声、去噪声。  
但这还是图像吧。视频为什么会因此更一致？

**苏格拉底：**  
别急。你还少了一个台阶。  
你以为扩散在什么空间里做这件事？

**我：**  
图像像素空间？一帧帧图片上加噪、去噪？

**苏格拉底：**  
这正是很多人的默认图景，也是第二个常见误解。  
现代高质量系统里，往往**不在像素空间直接扩散**，而是先把图像压到 latent 空间。

材料里举了个很具体的数：  
比如 Stable Diffusion 里，原图可能是 **256×256×3**，先通过 VAE 压成 **32×32×4**。  
这不是小修小补，是维度级别的缩减。

**我：**  
所以模型实际处理的不是图片本身，而是图片的压缩表示？

**苏格拉底：**  
对。  
先训练一个 VAE：编码器把图像压成 latent，解码器再把 latent 还原回图像。然后扩散模型真正加噪、去噪的对象，是这个 latent。

讲者说得很直白：  
如果你每一步都在 512×512 像素上做计算，本来就很贵，而扩散又是迭代过程，就更贵。  
所以 latent diffusion 真正解决的是：**把“概念上成立的扩散”变成“工程上能大规模训练的扩散”。**

**我：**  
也就是说，先压缩，再去噪，最后解码回去。  
那 Sora 的提升，核心是这个 latent 吗？

**苏格拉底：**  
latent 很重要，但它还不是主角。  
它像把原材料装进了更轻的箱子，让你搬得动。  
真正让 Sora 这条路线变得“越做越强”的，是接下来这一步：**把 latent 改写成 token。**

**我：**  
这就是 DiT？

**苏格拉底：**  
对。DiT，Diffusion Transformer。  
它不是简单粗暴地说“把 U-Net 换成 Transformer”。  
更准确地说，是：**把 latent 图像切成 patch，每个 patch 变成一个 token，再让 Transformer 去做噪声预测。**

讲者讲得非常像在拆 ViT：  
一张 latent 图像，被切成小块；每个小块展平成一个 token；加上位置编码；送进 Transformer。  
Transformer 输出后，再把 token 拼回去，得到每个 patch 上对应的噪声预测。

**我：**  
听起来像把图像当成一句话来处理。

**苏格拉底：**  
很像。  
但关键不是“像一句话”这个比喻本身，而是：**一旦问题被改写成 token 序列，Transformer 的规模优势就能接管。**

讲者还开玩笑说：“他们把 U-Net 换成 Transformer，发现——哇，居然很能扩展。显然，Transformer 本来就很能扩展。”  
这句玩笑其实很实在。真正的发现不是“Transformer 也能干这个”，而是“这个任务终于被放进了一个最吃规模的框架里”。

**我：**  
所以最核心不是 Transformer 三个字，而是 token 化，让它变成 Transformer 擅长的问题。

**苏格拉底：**  
你抓到骨头了。  
如果只说“用了 Transformer”，那像是在说把发动机品牌换了。  
但 DiT 真正在做的是：**重写问题的表示方式**。

这点在材料里有个很细的证据。讲者讲条件注入时，特地比较了几种做法：

- 最朴素的是把条件信息直接拼进序列，也就是 in-context concat；  
- 更常见的是用交叉注意力处理文本；  
- 再加上 adaptive layer norm 这类方式，把时间步、类别等小向量注入到每一层里。

他甚至说得很具体：直接把图像 token 和文本 token 混成一锅，“是个 mess”，效果并不好。  
这说明什么？说明**表示方式的组织**，不是边角工程，而是性能关键。

**我：**  
adaptive layer norm 那段我看得有点晕。  
它到底重要在哪？

**苏格拉底：**  
重要在“条件不是贴标签，而是改特征分布”。

普通 layer norm 做的事，是把一层里的特征拉回一个稳定分布。  
adaptive layer norm 则允许条件信息——比如时间步、类别——来决定平移和缩放参数。  
讲者给的直觉很到位：这相当于让条件直接决定，**哪些特征该被放大，哪些该被压低。**

这和把一个“truck”标签塞到输入尾巴上完全不是一回事。  
后者像你在纸边写了个注释；前者像你改了镜头的曝光和对焦。

**我：**  
好，那到这里我大概能串起来：  
扩散是在 latent 里预测噪声；DiT 把 latent 切成 token，用 Transformer 去预测；Transformer 又特别吃规模。  
所以“大模型 + 大数据 + Transformer”就把图像生成带上去了。

**苏格拉底：**  
还差半步。  
你说的是图像。Sora 的关键，是把这件事推到了视频。

**我：**  
就是把每一帧都切成 patch，然后拼起来？

**苏格拉底：**  
是，但你别小看这个“拼起来”。  
材料里讲者在 Sora 那部分最有价值的一点，就是他没有装作知道全部细节，而是说：公开信息有限，我们只能根据技术报告和 DiT 论文做合理重建。  
而这个重建里最关键的假设就是：**视频被压成时空 latent patch 序列，再交给 Transformer 建模。**

注意，不是“每帧各自生成，再拿胶水粘起来”。  
而是“整个视频被表示成一个统一的 token 场”。

**我：**  
这个差别为什么会大？

**苏格拉底：**  
因为视频最难的，不是每一帧画漂亮，而是跨帧关系。  
同一个人上一帧戴帽子，下一帧不能变成卷发；镜头左移时，背景和主体的相对运动要对；一个杯子不能转个身就变成花瓶。

如果模型只是逐帧看世界，它每次都在重新猜。  
如果模型处理的是统一的时空 token 序列，它至少有机会把“这是同一个对象”“这是同一个动作的延续”放进同一套关系里。

材料里讲者说得很朴素：“每个 token 代表的，是视频某个时空位置的一小块内容。”  
这句话不花哨，但很关键。  
一旦你接受这个表示，时间一致性就不再是后处理补丁，而是建模对象本身的一部分。

**我：**  
所以 Sora 震撼人的地方，不只是分辨率或质感，而是它看起来像在同一个序列里处理整段视频。

**苏格拉底：**  
对。  
你可以把它理解成一个很小但扎实的模型：

1. 扩散负责稳定的逐步去噪；  
2. latent 负责把问题压到可训练的表示空间；  
3. DiT 负责把表示改写成可扩展的 token 序列；  
4. Sora 最关键的延伸，是把 token 从二维图像推广到三维时空。

前面三步解释“为什么图像能做强”；第四步解释“为什么视频突然像跨了一代”。

**我：**  
但这样说会不会太乐观？  
好像只要“时空 token + Transformer”就万事大吉了。

**苏格拉底：**  
这就该问边界了。  
材料本身其实很克制。讲者反复说：我们**不知道** Sora 的全部工程细节，很多只能推测。  
所以不能把所有成功都草率归因到一个词上。

更实际地说，这条路线至少依赖几样东西：

- 规模：模型足够大；  
- 数据：而且是高质量、干净的数据；  
- 表示：时空 patch/token 的设计是否合理；  
- 条件：文本、图像、mask 等控制能否清晰注入；  
- 训练和采样工程：噪声调度、视频压缩网络、位置编码、推理策略等等。

讲者其实有点“兴奋但不装神”的味道：  
他说这大概就是“把 diffusion transformer 扩大很多，喂很多干净数据，再用海量算力训练”的结果。  
这话听着不性感，但很像真相的一部分。

**我：**  
你提到条件。材料里为什么还花那么多篇幅讲 prompt 重写、mask、视频编辑？  
不是主干架构更重要吗？

**苏格拉底：**  
这是很多人会忽略的地方。  
架构决定你“能不能做”；条件设计决定你“是不是按我想要的方式做”。

讲者在语言条件上抓了两个点：

第一，**更好的 caption 训练**。  
他提 DALL·E 3 时说，人类 prompt 常常太短、太糙，所以他们先让模型把图像重描述成更细的 caption，再用这些更高质量描述训练生成模型。

第二，**推理时的 prompt 扩写**。  
用户说“田野里的一只狗”，系统先把它展开成更长、更细的描述，再喂给模型。  
这是有偏置的，但讲者的态度很明确：**这个偏置看起来很有用。**

**我：**  
也就是说，模型“更懂 prompt”，可能不只是模型更聪明，也可能是因为你先帮它把人话翻译成了它更爱吃的话。

**苏格拉底：**  
正是。  
这点很像你给一个优秀的导演一句话梗概，和给他一份完整分镜，结果当然不一样。

**我：**  
那 mask 又是干什么的？

**苏格拉底：**  
mask 解决的是另一个问题：**哪些地方该保持，哪些地方该生成。**

讲者借 Lumiere 解释得很清楚。  
如果你给模型一帧图，想让它把后面几帧续出来；或者你已有一段视频，只想改其中某部分；那最笨的方法是让模型自己猜哪些区域该保留。  
更好的方法是直接给它：

- 已知内容；  
- 需要生成位置上的噪声；  
- 一个 mask，告诉它哪里要保留，哪里要续写。

这样一来，图生视频、视频延长、局部编辑，本质上都成了同一个问题：  
**在有明确约束的区域里做条件去噪。**

**我：**  
这听起来比“一个模型从零生成所有东西”更像工具，而不只是展示型 demo。

**苏格拉底：**  
你终于看见一个重要转向了。  
Sora 如果真的把这些能力整合起来，它强的不只是“随机抽一段很炫的视频”，而是更接近一个**统一的视频生成与编辑引擎**。

材料里讲者甚至说，他最感兴趣的部分，是让一个 policy 在 Sora 模拟的世界里玩 Minecraft。  
为什么这句话有意思？因为那已经不是“做一段好看视频”，而是把视频生成推向“可交互世界模型”的边上了。

当然，这里要小心：这更像启发，不是已被证明的能力边界。

**我：**  
我现在有点明白为什么讲者那么兴奋了。  
他兴奋的不是某个神秘技巧，而是：看上去像魔法的东西，其实可以顺着一条连续技术线理解。

**苏格拉底：**  
是。  
这份材料最好的地方，不是它宣称“我知道 Sora 的秘密”，而是它在做一件更诚实的事：  
**用 DiT 搭桥，把 Sora 从神迹还原成一条技术演化路径。**

从基础扩散，到 latent diffusion，到 DiT，再到视频时空 token。  
这条路径一旦成立，Sora 的“跳变感”就会变得没那么玄学。

**我：**  
但如果我下次再看到一个新模型，说自己也是什么“时空 token + Transformer + 扩散”，我该先检查什么？  
不然很容易被术语唬住。

好问题。先别检查 marketing，先检查三个变量。

**第一，看它建模的单位到底是什么。**  
是逐帧独立，还是统一的时空表示？  
如果它本质上仍是“一帧一帧补”，那一致性上限通常会受限。

**第二，看条件是怎么进来的。**  
文本只是随手拼进去，还是有结构化注入？  
有没有 prompt 重写、交叉注意力、mask 这类让控制真正生效的机制？  
很多模型不是不会生成，而是不会“按要求生成”。

**第三，看它是否真的吃规模。**  
不是嘴上说 Transformer，而是看表示、训练目标和架构设计是否让加大模型、数据、算力后能稳定变强。  
DiT 重要，不是因为它“长得像 Transformer”，而是因为它在实验里表现出明确的可扩展性。

**我：**  
所以最脆弱的前提，其实不是“有没有 Transformer”，而是“这个问题是不是被改写成了 Transformer 能稳定扩展的问题”。

对。  
这是你该带走的最小模型。

**Sora 的关键，不是把视频交给更大的模型，而是把视频先表示成一个适合规模化建模的时空 token 序列，再用扩散去噪的方式在这个序列上维持结构。**

如果这个表示对了，规模、数据、算力才会像顺风。

那材料里那个“只对 latent 前 3 个通道做 guidance，效果跟 4 个差不多”的细节，算什么？

它小，但很说明问题。

有些通道更承担语义控制，有些通道可能承担别的统计结构。  
这说明 latent 不是一团均匀的泥，而是内部有分工。

不过这类点，在这份材料里还只是边角启发，不该被拔高成主因。

对。  
别把附录里的趣味细节，误当成发动机。

这个问题，材料没有替你答完。
