LTX-2 × Audiovisual Generation

LTX-2：联合音视频生成的最小心智模型

LTX-2 把声音和画面放进同一次生成，不把音频留到最后再补。理解它要抓住三件事：表示分开、生成耦合、时间对齐。

pinyu 2026.05 论文
核心问题

视频和音频经常来自同一个事件，但它们不是同一种信号。视频是时空结构，音频是时间结构；两者相关，却不对称。

LTX-2 没有把两种模态揉成一种东西。它让它们留在各自合适的表示里，再共同生成同一个事件。

00

为什么不能后配音

很多声画关系，靠后期补一条音轨并不能真正补齐

先抓住这篇论文真正要解决的矛盾：视频和音频经常来自同一个事件，但它们不是同一种信号。

如果先生成视频再补音频，声音只能给既定画面“配合”。如果先生成音频再补视频，画面又只能倒推声音背后的场景。一个人说话时，嘴型受发音约束；玻璃碎裂时，声音要和碎裂时刻对上；车从左向右开过时，画面里的运动和声音的空间变化也应该一起发生。

这些不是“视频 + 一条声音轨”。它们更像同一个物理事件在两个感官里的投影。

串联式

video first -> audio later
声音只能给既定画面补细节。

vs
LTX-2

text -> generate video and audio together
声音和画面共同生成同一个事件。

更正式一点，它想学的是 p(video, audio | text)，也就是在文本条件下，视频和音频的联合分布。

01

为什么先分开表示

联合生成不等于把所有东西塞进同一个 latent

视频是 3D 时空信号：有横向、纵向、时间，还要处理物体、纹理、运动、镜头。音频更像 1D 时间信号：重点是节奏、频率、语音、环境声。

如果硬塞进同一个 latent，位置编码、压缩率、模型容量都会变别扭。LTX-2 的做法更克制：先保留两种模态最自然的表示，再让生成过程发生强耦合。

Video path

视频先经过自己的 video VAE，变成 video latent。它需要保留空间结构、运动轨迹和镜头变化，所以 token 更大、信息密度更高。

关键词：space / motion / scene layout

Audio path

音频先经过自己的 audio VAE，变成 audio latent。它更关注时间节奏、频率结构、语音和环境声，所以 token 更小、更偏时间组织。

关键词：rhythm / speech / ambience
关键句
真正重要的是这八个字：表示分开，生成过程耦合。
02

每一层里到底在做什么

双流结构的重点，在每层都反复对齐，不只是两条路并排跑

LTX-2 的核心是一个不对称双流 Diffusion Transformer。你可以把它想成两条并行的生成流：一条管视频，一条管音频。每一层里，两条流都按相似顺序工作。

Current tokens
Self-attention
Text cross-attention
AV cross-attention
FFN
Updated tokens

Self-attention 回答的是“我现在生成到什么状态了”。Text cross-attention 回答的是“prompt 要我往哪里去”。Audio-video cross-attention 回答的是“另一边此刻发生了什么，我要怎么同步”。

这一层的顺序很重要。模型会先确认自身状态，再回到文本目标，最后根据另一模态校准结果。

03

为什么两边都要看文本

文本负责意图，跨模态负责对齐，这两个工作不能互相替代

这里最容易被跳过、但其实很关键的一步是：为什么音频和视频要各自看文本？

Video reads text

视频看文本，是为了知道画面里该有谁、在哪里、做什么、是什么风格。

scene / character / action / style

Audio reads text

音频看文本，是为了知道该说什么语言、什么语气、什么台词，以及有什么环境声。

speech / language / emotion / ambience

另一模态不能可靠替代文本完成这件事。视频不能告诉音频“具体台词是什么”；音频也不能告诉视频“场景是不是空旷车库”。所以每一层都需要先把各自拉回 prompt，再去互相校准。

Compression
text sets the goal
cross-modal attention aligns the result
04

音频和视频怎么互相读

所谓“互相读”，说的就是双向 cross-attention

互相读，技术上就是双向 cross-attention。音频流读取视频证据，视频流读取音频证据。这种交换不会只发生在最后一步；每一层都会互相看一眼。

Video -> Audio

音频流读取画面证据，比如嘴部形状、车辆运动、脚步节奏、空间大小和远近关系。

Audio -> Video

视频流读取声音证据，比如语音节奏、发音结构、撞击发生的时刻，以及声音何时开始衰减。

Cross-Modal Update
reader_update = attention(Q_reader, K_source, V_source)
reader = reader + gate * reader_update

reader 是正在被更新的一边，source 是被读取的一边。音频读视频时，audio 是 reader，video 是 source；视频读音频时则反过来。

不要误读
这里不是单边服从。更像是两边在生成过程中持续互相约束，避免各写各的。
05

公共时间轴怎么对齐

它靠的是时间上的软对齐：先在同一时刻相遇，再在内容上选择对象

关键缺口在这里：音频和视频 token 数量不同、结构不同，怎么知道该读哪个时间附近的东西？

LTX-2 没有用硬规则规定“第 37 个音频 token 必须对第 37 帧视频”。它采用的是更柔性的时间对齐：给两边 token 都挂上时间位置，然后在跨模态 attention 里只使用时间位置。

Shared Temporal Axis
video token: (x, y, t)
audio token: (t)

cross-modal attention: only use t for positional alignment
score = Q_audio(t_audio) · K_video(t_video)
Video token

视频内部仍然有空间位置 x, y 和时间 t。它保留场景布局和视觉事件的空间分布。

Audio token

音频内部主要是时间 t。它关注语音、撞击、环境声在时间维上的组织。

Cross-modal

跨模态时只用时间位置做相遇条件，让相近时间的 token 更容易互相注意。

  1. 时间先缩小范围。 第 3 秒附近的音频 token 更容易去读第 3 秒附近的视频 token。
  2. 内容再选择对象。 在这些视频 token 里，最终读嘴、车、脚步还是玻璃，交给 attention 的内容匹配来决定。
  3. Gate 决定混入多少。 早期更重整体结构，后期再细化嘴型、撞击声和空间声。

时间负责让两边在同一时刻相遇，attention 负责选择具体对象，gate 负责决定混入多少。

06

一个具体场景

把抽象结构压回一个能感受到的例子里

Prompt

一个人在空旷车库里说话，远处一辆车从左向右驶过。

串联式方案

先生成视频，再让音频模型去猜：这个空间多空旷？车声该怎么移动？说话有没有混响？嘴型和发音是否对得上？

vs
LTX-2

视频流先知道要生成车库、人物、嘴部和车辆运动；音频流先知道要生成语音、混响、车声和环境感。然后两条流在每一层互相读，持续同步。

所以，“音频知道车在移动”并不是显式推理。更像是音频 token 在对应时间段读到了车辆运动相关的视频 token。“视频受语音节奏约束”也不是后处理；它来自嘴部附近的视频 token 读到了音频里的语音节奏和发音结构。

它真正要解决的是：声音和画面是否像同一件事。至于“有没有声音”，反倒是更浅一层的问题。

07

怎么迁移使用

这篇论文真正值得带走的，是一个系统设计判断规则

如果以后看多模态系统，可以抓住一个判断标准：

当两个模态结构不同，但描述的是同一个过程时，不要急着共享表示；先保留各自最自然的表示，再找出真正需要强对齐的公共轴。

在 LTX-2 里，这个公共轴是时间。这比“多模态就把 token 拼一起”更精确。拼 token 只是把东西放在一起；联合生成要求它们在生成过程中互相约束。

  • 模态天然异构时，先想清楚它们各自最适合什么表示，再谈能否统一成一个 latent。
  • 如果它们描述的是同一个过程，再问真正需要强对齐的共享轴是什么，可能是时间，也可能是动作意图或空间事件。
  • 如果共享轴找错了，系统就容易出现“形式联合、实质脱节”的问题。
08

最容易误读的地方与它的边界

它的聪明之处在克制；它的短板也正来自这个作用范围

LTX-2 的聪明之处，不在于追求“大一统”。它先承认音频和视频本来就不一样。它没有把两者硬塞进同一个 latent，也没有追求完全对称的双塔，只在必要处强耦合。

最值得迁移的工程思想
该分开的地方分开，该同步的地方高频同步。

它强在局部事件级同步：谁在说话、什么时候撞击、声音是否跟画面动作对上、空间感是否匹配。

边界
它弱在长程因果一致性。超过约 20 秒、多说话人、低资源语言、复杂叙事关系时，模型仍然容易漂。因为那时问题已经超出了音视频对齐本身，开始牵涉角色、记忆、世界状态和故事因果。

换句话说，LTX-2 解决的是“同一时刻的事件是否一致”，还没有完全解决“跨很长时间的故事是否一致”。

迁移题
如果要做“机器人动作 + 摄像头视觉 + 触觉传感”的联合生成，你真正该强共享的轴，是空间、时间、动作意图，还是别的东西？
一句带走

LTX-2 让声音和画面在各自合适的表示里，共同生成同一个事件；它不是在视频模型外面再挂一层音频。

先保留模态差异，再找真正需要强对齐的公共轴，这比一上来追求统一表示更重要。
