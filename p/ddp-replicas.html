Private Replicas, Identical Update

Data-parallel training does not put one model object on many GPUs. Each process keeps a private replica, and collective communication makes those replicas take the same update.

pinyu 2026.07 torch.distributed
The idea

Two ordinary Python processes have separate memory. Each starts with its own copy of a model parameter. How can they train as if they were updating one model?

Not by sharing memory. By sharing a schedule. Each process keeps a private replica. At the gradient boundary, every process joins a collective that leaves it with the same reduced gradient. Then each process updates its own weights with that common value.

Private process state plus a shared communication contract is not the same thing as a shared Python object.

1

Two processes, one update

Same script, two processes, different local data — and both end with the same private parameter.

A minimal CPU program is enough. No GPU is required. The example uses Gloo so it runs on a laptop; real multi-GPU jobs usually pick NCCL instead. The shape of the story is the same either way.

toy_ddp.py 
import torch
import torch.distributed as dist

dist.init_process_group(backend="gloo")
rank = dist.get_rank()
world_size = dist.get_world_size()

# Each process owns a private model replica.
# They start equal only to keep the example small.
weight = torch.tensor([1.0])

# Different local data → different local gradients.
local_grad = torch.tensor([2.0 if rank == 0 else 6.0])

print(
    f"rank={rank}: weight={weight.item()}, local_grad={local_grad.item()}",
    flush=True,
)

# Sum gradients from every rank into every rank's local tensor.
dist.all_reduce(local_grad, op=dist.ReduceOp.SUM)

# DDP normally averages gradients.
mean_grad = local_grad / world_size
weight -= 0.1 * mean_grad

print(
    f"rank={rank}: mean_grad={mean_grad.item()}, updated_weight={weight.item()}",
    flush=True,
)

dist.destroy_process_group()

Launch it with two processes:

shell 
torchrun --standalone --nproc-per-node=2 toy_ddp.py

The logical result looks like this (print order is nondeterministic):

rank=0: weight=1.0, local_grad=2.0
rank=1: weight=1.0, local_grad=6.0
rank=0: mean_grad=4.0, updated_weight=0.6
rank=1: mean_grad=4.0, updated_weight=0.6

Nothing was shared in memory. No process handed model ownership to another. No parameter server sat between them. Both private copies still moved from 1.0 to 0.6.

That is the mystery. The rest of the article tracks what makes it possible.

2

The goal is not sharing a model

The job already has two replicas. What it needs is agreement, not shared ownership.

A data-parallel job starts with private state on both sides:

Rank 0 · private

weight = 1.0
local gradient = 2.0

vs
Rank 1 · private

weight = 1.0
local gradient = 6.0

If each process stepped on its own gradient alone, the replicas would diverge. Rank 0 would move one way, rank 1 another, and after a few steps they would no longer be the same model in any useful sense.

A common assumption is that the GPUs must somehow share one model object. They do not. Each process holds a replica, and the job keeps those replicas aligned by synchronizing the update they apply.

The real problem

Not: how can both processes reach one shared object?

But: how can both replicas agree on one update?

3

torchrun starts processes and gives them coordinates

A launcher is not a collective. It is how workers get identities and a way to find one another.

torchrun is a process launcher. In the simplest local case it starts worker processes for your training script, assigns each worker an identity, injects environment variables such as RANK, LOCAL_RANK, and WORLD_SIZE, and provides the rendezvous configuration the workers need to discover one another.

RANK

Global identity in the whole job. Rank 0 often owns logs or checkpoint writes, but for collectives every rank is a peer.

LOCAL_RANK

Identity on this machine. On a GPU job you usually pin work to cuda:LOCAL_RANK.

WORLD_SIZE

How many processes participate. Two ranks on one node means WORLD_SIZE=2.

torchrun is useful, not magical. You can spawn two Python processes yourself, export the same environment variables, point them at a valid rendezvous endpoint, and still form a working job. What torchrun buys you is reliable bookkeeping for that setup — especially across many machines.

What torchrun does not do
It does not move gradients, create a process group, or enable DDP. It starts processes and coordinates how they discover each other.
4

init_process_group turns coordinates into a group

Until this call, each process is isolated Python. After it, each process belongs to a communication group.

Each process begins alone: private address space, private tensors, no agreed peers. dist.init_process_group() is the rendezvous. Using the launch configuration, every process finds the others and initializes the default process group.

That call has a precise meaning. Each process still keeps its own Python state and its own distributed-library state. Each receives a local handle to the same logical communication group; calls that omit group=... use that default group. And init_process_group() itself does not shard a model, synchronize gradients, or enable DDP.

After init, two isolated runtimes are members of one process group. The boundary is group membership, not a shared heap.

5

all_reduce is the operation that keeps replicas equal

Every rank gives a local tensor. Every rank ends with the same reduced result — still in private memory.

The numbers are small enough to check by hand.

Before all_reduce:

rank 0 owns local_grad = 2
rank 1 owns local_grad = 6

After all_reduce(SUM):

rank 0 owns local_grad = 8
rank 1 owns local_grad = 8

After dividing by world size:

both own mean_grad = 4

After the optimizer-like step weight -= 0.1 * mean_grad:

both privately update weight from 1.0 → 0.6
What all_reduce does

all_reduce(SUM) reduces every rank's contribution and writes the same reduced value into each rank's own local buffer.

Average those synchronized gradients and both replicas take the same step. That is the core idea behind data-parallel training: DDP does not make many GPUs hold one model. It makes many model replicas take the same gradient step.

What the toy covers
The example captures the semantic core: synchronize, average, update the local replica. Real DDP also installs autograd hooks, keeps parameters aligned at construction, buckets gradients, and often overlaps communication with backward. This is a teaching model, not a reimplementation.
6

A collective is a contract, not shared memory

Every participant must enter compatible collective calls in the same order.

An all_reduce is not "rank 0 asks rank 1 for data." It is not shared memory, and it is not an ordinary RPC that one process can fire alone and forget.

A collective is a group-wide protocol step. Every participating rank must eventually enter compatible collective calls in the same order. They do not need to hit the same wall-clock instant. What must line up is group membership, operations, tensor shapes, and call order.

Anti-pattern
Rank 0 enters a collective that expects the whole group. Rank 1 never participates. Do not run this fragment without a short timeout.
broken.py · conceptual 
if rank == 0:
    dist.all_reduce(local_grad)  # rank 1 never arrives

Depending on backend, timeouts, and configuration, the failure may surface as a timeout, a communication error, or an apparent hang. Not every broken collective is an infinite hang, but mismatched participation is always a hard correctness bug.

Break the contract

Simulation only — not a live hang. Pull one rank out of the collective and see the group fail to finish.

The rule

all_reduce is not "ask another process for data." It means: every member of this group performs the next agreed protocol step.

Compatible shape, dtype, operation, and order all matter. A mismatched collective sequence is enough to break the job even when each local model's math is fine.

7

DDP is this schedule, automated

The manual toy and real DDP share one plan: synchronize gradients, then update local replicas.

Stage Manual toy DDP
Local forward / loss Implicit (fixed local_grad) Ordinary autograd on a local mini-batch
Local gradient Constructed by hand Autograd fills .grad
Synchronize dist.all_reduce, then average DDP hooks reduce (and average) during backward
Update weight -= lr * mean_grad on each rank Optimizer steps the local replica on each rank

The optimizer still runs locally on every rank. Nothing ships model weights out of process just to take a step. Replicas stay aligned because the gradients they step on have already been equalized by the collective schedule.

In short
Autograd computes local gradients. Collectives synchronize those gradients during backward. Each optimizer then updates its own local replica with the same values.
8

What changes across machines — and what does not

Launch coordinates change. The meaning of a collective does not.

A two-node job still runs the same training script. Each node starts one torchrun, which launches its local workers. The master address is for setup and rendezvous, not a permanent parameter server on the data path.

node 0 
torchrun \
  --nnodes=2 \
  --node-rank=0 \
  --nproc-per-node=2 \
  --master-addr=10.0.0.1 \
  --master-port=29500 \
  toy_ddp.py
node 1 
torchrun \
  --nnodes=2 \
  --node-rank=1 \
  --nproc-per-node=2 \
  --master-addr=10.0.0.1 \
  --master-port=29500 \
  toy_ddp.py

One machine or many?

Rendezvous is setup only. Collectives belong to the worker group.

Across machines you still have private replicas, environment coordinates, a process group, and the same collective contract. Placement changes. Meaning does not.

9

What to keep

The decision rule

torchrun assigns identities and rendezvous coordinates. init_process_group turns those coordinates into group membership. Collectives then synchronize private state under a shared contract, and DDP places that synchronization at the gradient boundary.

Private replicas. Identical update. No shared Python object required.

Scope
Private state plus ordered group protocols is the foundation for later systems such as FSDP, tensor parallel layouts, and device meshes. Those change what is partitioned. The contract still rules how ranks agree.
